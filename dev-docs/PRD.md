# Quizbun — Product & Architecture Reference (v1.0)

Status: **v1.0.0 — released, the Standard is frozen** · Last updated: 2026-07-06

This is the single, self-contained reference for what Quizbun is and how it is built at v1.0. It is the starting point for future development: it does not depend on any other document to be understood. It consolidates what were previously three separate artifacts — the product-requirements document, the per-milestone build plans, and the architecture decision records. Those described *how the project was going to be built, in what order*; this one describes *what exists now and why it was built that way*.

A few living documents remain the executable or normative authority for exact rules, and this reference summarizes rather than replaces them:

- `src/shared/lib/quiz/schema.ts` — the Zod schema, the executable single source of truth for validation.
- `docs/standard.md` — the normative Quiz Object Standard, written for authors and Renderer implementers.
- `CONTEXT.md` — the binding glossary for the capitalized domain terms used throughout (Quiz, Question, Option, Run, Library, Catalog, Explanation, Content hash, Page size, Voice, …). Terms are used here with their glossary meaning.
- `docs/idea.md` — the long-form product rationale.

Where this document summarizes a rule, the artifacts above win on any conflict of detail. Everything the plans and ADRs recorded that has no other home — architectural decisions, rejected alternatives, and resolved micro-decisions — is folded in below in full.

> [!NOTE]
> **On visual design.** v1.0's scope was correctness, structure, and the end-to-end learning loop — not visual polish. The structural baseline is required and in place: semantic HTML, full keyboard operability, and responsive layout. A token-based CSS system exists (`src/shared/styles/vars/`), but visual design is the phase that follows v1.0, not part of it. Do not read the absence of polish as a defect in v1.0 scope.

## 1. What Quizbun is

Quizbun is a static, explanation-first quiz catalog for self-learners, built around a versioned JSON quiz format — the **Quiz Object Standard** — designed to be generated by AI. It is open source (MIT). "Static" is a runtime constraint, not a codebase constraint: the built site runs entirely in the browser with no database, no API, and no servers of any kind. The frontend itself may be as ambitious as needed.

**The core loop the whole product is shaped around:** ask an AI tool to generate a quiz using the published prompt and schema → copy the JSON from the chat → paste it into Quizbun → learn with explanation-first feedback → resume, retake, reset. Every design decision keeps that loop short and reliable.

The product is both a learning site and a reusable content standard: the same Quiz JSON can be AI-generated, validated automatically, reviewed in pull requests, and rendered by different layouts in the future.

**Product principles** (these explain why the architecture is shaped the way it is):

1. **Learning before testing.** Every Question teaches. After submission the Explanation always shows, correct or not. Results are binary — correct or incorrect — because the reward is the Explanation, not a score.
2. **Static by default, local by choice.** Public quizzes are built into the site at build time. A user's own quizzes are imported and stored locally in the browser. Nothing ever leaves the device.
3. **One standard, AI-first.** The Standard is designed for machine generation: minimal required fields, hard defaults instead of configuration flags, strict validation that turns generation mistakes into actionable errors. Every removable field is one less way for generated JSON to be invalid.
4. **Markdown only, rendered safely.** Quiz content is Markdown, never raw HTML. HTML in source is stripped, and rendered output is sanitized as defense-in-depth.
5. **Open contribution model.** The Catalog is open for pull requests. Structure and schema are checked by CI; Explanation quality and factual correctness are reviewed by humans. Submitting a public quiz licenses it under the repository's license.
6. **Presentation-neutral content.** The Quiz object describes content and correctness, never how it looks. Zero presentation fields — no pagination, no Option letters, no themes. A useful invariant: Questions carry no configuration that changes how a result is *interpreted*, only validation config that defines what *counts* as correct. This keeps all Renderers trivially compatible.

## 2. Users

- **Self-learner** — takes quizzes; the only user the runtime serves. Phone-first for the player.
- **Creator** — generates quizzes (typically with AI) for private use; lives on the import page and the docs.
- **Contributor** — submits a quiz to the Catalog via pull request; lives in the repo and CI.

One person is usually all three. There are no accounts and no identity anywhere in the product.

## 3. The Quiz Object Standard (frozen at v1)

The Standard is the core artifact. A Quiz is one JSON object: metadata plus an ordered list of Questions. The Zod schema (`src/shared/lib/quiz/schema.ts`) is the single source of truth; the JSON Schema at `/schema/quiz.v1.json` is generated from it. This section summarizes the frozen v1 shape; `docs/standard.md` is the normative text.

### Versioning and strictness

- `schemaVersion` is the integer `1`. Strings such as `"1"` or `"1.0"` are invalid.
- Validation is **strict at every level** — unknown fields are errors on the Quiz, Question, Option, and validation objects. When authors are AIs, leniency is where errors hide; a hallucinated or misspelled field must fail loudly at import.
- **Error messages are a product feature.** They are path-precise and actionable enough to paste back into an AI chat to get a corrected file. The same formatter (`src/shared/lib/quiz/format-errors.ts`) drives the import page and the CI validators.
- Within version 1, only **adding optional fields** is allowed. Removing a field, changing a required field, tightening a constraint, or changing correctness semantics requires `schemaVersion: 2`. A Quiz that validates today keeps validating.

### Quiz-level fields

| Field           | Required | Notes                                                               |
| --------------- | -------- | ------------------------------------------------------------------- |
| `schemaVersion` | yes      | Integer `1`.                                                        |
| `id`            | yes      | Kebab-case slug (lowercase latin letters, digits, single hyphens).  |
| `title`         | yes      | Short (inline-only) Markdown field; non-empty after trimming.       |
| `questions`     | yes      | Ordered array, at least one; Question `id`s unique within the Quiz. |
| `description`   | no       | Long Markdown field; non-empty when present.                        |
| `language`      | no       | BCP-47-shaped tag (`en`, `en-US`).                                  |
| `tags`          | no       | Array of Tags; defaults to `[]`; same charset as `id`.              |
| `author`        | no       | Free-form string; non-empty when present. Not an account.           |

Quiz `id` is the Library primary key. Library and Catalog are **separate namespaces**: a private Quiz may reuse a Catalog `id` freely.

### Questions

Every Question has `id` (unique within the Quiz, kebab-case), `title` (short Markdown prompt — the question itself, phrased as the actual ask), `type`, and `explanation` (long Markdown, shown after submission regardless of correctness). Optional: `description` (long Markdown supporting context rendered as secondary text — a scenario, code, constraints; never the question itself) and `references` (long Markdown for links/citations, shown after the Explanation); both must be non-empty when present. Question `id` is part of Progress identity — changing it creates a new Question from the storage perspective.

Three types, all in v1: **`single-choice`**, **`multiple-choice`**, **`input`**.

**Choice Questions** use an `options` array of bare `{ text, isCorrect }` objects — no ids, no labels. Rules:

- At least two Options.
- Option identity is its index in the **original JSON order**.
- `single-choice`: exactly one Option with `isCorrect: true`.
- `multiple-choice`: at least one correct Option; all Options correct is valid. All-or-nothing — the submitted original-order index set must exactly match the correct set. No partial credit.

**Input Questions** use a `validation` object with hard defaults so the common case needs zero flags:

- `mode: "text"` — compares against every `acceptedAnswers` string; trims, collapses internal whitespace to one space, normalizes to Unicode NFC, and is case-insensitive unless `caseSensitive: true`.
- `mode: "numeric"` — `acceptedAnswers` are finite JSON numbers only (numeric strings are invalid); optional `tolerance` (default `0`); accepts `.` or `,` as the decimal separator (not both in one value); no thousands separators; correct when `abs(submitted − accepted) <= tolerance` for any accepted answer.

### Correctness model

Every Question evaluates to exactly one binary result. No points, weighting, partial credit, scoring configuration, or attempt history in v1. A Summary reports `X of Y correct`. A Renderer may block empty submissions for usability; because every choice Question has at least one correct Option, an empty selection can never be correct.

### Content formatting (two-tier Markdown)

Every text field is Markdown, rendered safely; raw HTML is always stripped. Two tiers:

- **Short fields** (Quiz `title`, Question `title`, Option `text`) are **inline-only**. Block Markdown in a short field is flattened/degraded at render time, never rejected at import.
- **Long fields** (Quiz/Question `description`, `explanation`, `references`) allow **full Markdown**, including fenced code blocks with syntax highlighting for a fixed language set: JavaScript/TypeScript family (`js`, `ts`, `jsx`, `tsx`), `json`, `html`, `css`, Python (`py`, `python`), Bash/shell (`bash`, `sh`, `shell`), and `sql`. Any other or missing fence info renders un-highlighted — never an error. Highlighting is Renderer presentation and adds no fields to the Standard.

### Two normative Renderer rules

These preserve content identity and are part of the Standard, not the app:

1. **Saved choice answers reference original-order Option indexes**, never the displayed order. A Renderer that shuffles must translate displayed positions back to original indexes before saving or checking.
2. **Progress is keyed by Quiz `id` + Question `id`, invalidated per Question by a Content hash.** If a Question's content changes on re-import, that Question's saved answer is discarded while unchanged Questions keep Progress.

Shuffling, Option labels, pagination, Page size, keyboard controls, and layout are all Renderer behavior and must never be written into a Quiz object.

### Resolved micro-decisions (frozen into v1)

These were the small gaps the design interviews never pinned down; they are now normative and changing any is a `schemaVersion: 2` break:

1. Choice Questions require **at least two** Options — one Option is not a choice.
2. `multiple-choice` requires **at least one** correct Option; all-correct is valid. "Select none is correct" is not expressible in v1.
3. **No hard length caps** in v1 (title, Option text, Tag count, Question count) beyond non-empty strings and required non-empty arrays. Loosening is additive; tightening is breaking — so v1 defaults to loose rather than guessing caps.
4. Numeric `acceptedAnswers` are **JSON numbers only**, since JSON already has numbers.

### Published artifacts and the Public catalog profile

- The **JSON Schema** at `/schema/quiz.v1.json` is generated via `z.toJSONSchema()`. It is necessarily looser than Zod (cross-field rules like single-choice correctness become prose annotations); the import page and Zod validator are the final authority. CI regenerates it and fails on any drift against the committed file.
- The **AI generation prompt** (`docs/quiz-generation-prompt.md`) embeds the JSON Schema plus one canonical example and instructs "output one JSON object, nothing else."
- **Canonical examples** in `docs/examples/` (one per type plus the numeric-input variant) are validated in CI by the same Zod schema.
- The **Public catalog profile** is a stricter rule set applied **only in CI** to repository content — a profile on top of the Standard, never a second schema. It requires `description`, `language`, and at least one Tag, plus repo-wide `id` uniqueness and filename = `id`.

## 4. Architecture and code layout

**Stack:** Astro 7 static output with React 19 islands, deployed to GitHub Pages. Bun is the package manager (Node ≥ 22.12). No backend. JS islands hydrate only where interactivity genuinely exists (player, import, Library, tag filter, continue block, copy-prompt); build-time-known content is never client-side rendered.

**Base path:** GitHub Pages builds set `GITHUB_PAGES=true`, which switches the Astro `base` to `/quizbun` and sets `site`. Never hardcode absolute paths to site routes or assets — always route through the base-aware helpers (`withBase` and the docs loader's link rewriting).

**Code layout follows Feature-Sliced Design.** Current layers and their live slices:

- `src/app/` — the app shell: layouts (`layout.astro` root shell, `docs-layout.astro`) plus the every-page chrome they mount (`ui/` header, theme switcher, footer, voice picker; `lib/` theme preference). Invariant: app chrome is referenced only by app layouts — the first consumer outside `app/` demotes the component to a then-recreated widgets layer.
- `src/pages/` — **only** Astro route files (`.astro` pages plus the few endpoints that must be `.ts`, e.g. `docs/examples/[file].ts`). No React components or page-private glue live here. Routes are the hydration orchestrators: they compose page-slice parts and assign every `client:` directive.
- `src/_pages/` — the FSD pages layer (routes + slices together form it): per-screen React compositions named after the URL surface: `home` (marketing sections), `quizzes` (Catalog list + filter), `quiz` (detail ↔ player), `library` (Library list + filter), `library-quiz` (detail ↔ player). A page slice exists only when a route needs page-specific React/model code — thin routes keep mounting features/shared directly. Page slices never import each other (the two quiz screens each carry their own copy of the small `render-player` glue by design) and never export a whole-page component that would force all-or-nothing hydration.
- `src/features/` — user-facing capabilities: `player`, `import-quiz`, `quiz-detail`, `filter-by-tags`, `continue-runs`, `copy-prompt`.
- `src/entities/` — `quiz` (the Quiz card / list-item presentation of metadata).
- `src/shared/` — cross-cutting building blocks. `shared/lib/` homes: `quiz` (schema, types, Content hash, error formatter), `storage` (IndexedDB + preferences), `render` (Markdown), `content` (build-time public-quiz loader), `docs` (build-time docs loader), `speech` (Read aloud store), `routing`, and `errors`; plus `shared/styles/` and `shared/ui/`.

There is **no `widgets/` layer**, recreate it only when a true widget appears — a reusable block with limited business logic that composes features/entities and is not itself a page.

### Storage

- **IndexedDB** holds quizzes and Runs. One database, promise-wrapped, with a versioned `onupgradeneeded` migration scaffold. Private quizzes are stored as an envelope `{ quiz, importedAt }` so the bare Quiz object stays presentation- and storage-neutral (Export emits only the Quiz, never the envelope or Progress).
- **Exactly one saved Run per quiz** by construction. The Run records, per Question keyed by Question `id`, `{ contentHash, submittedAnswer, isCorrect }`, plus `startedAt`, optional `finishedAt`, and `updatedAt` (bumped on every answer save). `updatedAt` was added additively — readers fall back to `startedAt` for older Runs, with no schema version bump.
- **Run namespaces.** The `runs` store key is `` `${source}:${quizId}` `` where `source` is `catalog` or `library`. This lets a public quiz and a re-imported same-`id` private copy track Progress independently in one store with one query path.
- **Content hash** (`src/shared/lib/quiz/content-hash.ts`) is a deterministic SHA-256 (via `crypto.subtle`) over a stable serialization of a Question. Any change to prompt, Option text, Option order, accepted answers, or validation mode changes the hash, which invalidates that Question's saved answer on re-import.
- **Progress reconciliation on replace:** re-importing a Quiz with an existing `id` offers replace-or-cancel; replace keeps a saved answer only where a Question with the same `id` exists *and* its Content hash still matches, and discards the rest.
- **localStorage** holds UI preferences: **Page size** (`1 | 3 | 5 | 10`, default `5`, global — not per quiz) and the Read-aloud **Voice** URI (`quizbun.voice-uri`). Reads are tolerant — an invalid stored value falls back to the default.

### Markdown rendering

Two-tier safe rendering via `marked` + `sanitize-html` (`src/shared/lib/render/`): `renderInlineMarkdown` for short fields (inline tokens only, safe inside a `<label>` or list item) and the full renderer for long fields. Raw HTML is stripped in both tiers; hostile input (script tags, event handlers, `javascript:` links) is stripped. A single field→tier mapping mirrors `docs/standard.md`. For contexts that cannot render Markdown (browser tab title, compact lists), formatting is flattened to plain text.

### Decision: query-param routes for private quizzes

Private Library quizzes exist only in the visitor's IndexedDB, so their ids are unknown at build time — and GitHub Pages serves only files that exist at build time. Private quizzes are therefore addressed through a prebuilt **static shell plus a query param**: `/library/quiz/?id={id}`.

Rejected alternatives:

- **`404.html` fallback hack** for pretty paths like `/library/{id}/`: every deep link physically returns 404 before a script rewrites it — a flash of error page, and fragile under a project base path.
- **Hash routing** (`/library/#/quiz/{id}`): works, but imports SPA-router machinery into an otherwise static site.

A private quiz URL is only meaningful on the device that holds the quiz, so its reduced "prettiness" costs nothing real. Do not "upgrade" these routes to path-based URLs without first revisiting the GitHub Pages constraint.

### Decision: docs pages rendered from repo Markdown via a build-time loader

The site's `/docs/…` pages are rendered at build time from the canonical repo Markdown (`docs/standard.md`, the AI prompt, the examples README, `docs/contributing.md`) by a small loader in `src/shared/lib/docs/` — **not** by Astro content collections, and never by keeping a second copy of the text.

- **Single source of truth:** the repo file is canonical; the site page has no content of its own and cannot drift.
- **Link rewriting** happens on marked's token tree: links between published docs become base-aware site routes; the committed schema artifact maps to `/schema/quiz.v1.json`; canonical example JSONs map to `/docs/examples/{file}` (served by a static endpoint reading `docs/examples/` directly); links to repo-only files (e.g. `CONTEXT.md`) point at GitHub; absolute live-site URLs are rewritten base-relative so previews work.
- **Build-time dead-link check:** a docs link that resolves to neither a site page nor an existing repo file throws inside the loader and fails `bun run build`.
- **Rendering** preserves the full heading hierarchy (unlike the quiz tiers), degrades GitHub `> [!NOTE]`/`[!IMPORTANT]` alert markers to bold labels, and strips raw HTML as everywhere else.

Astro content collections were rejected because the sources must stay readable and linkable on GitHub under `docs/`, and the only transform needed is link rewriting — which collections do not provide. Revisit only if the docs section grows past a handful of pages.

### Decision: Read aloud via the browser's on-device speech engine

The Renderer can read a Question's Explanation aloud as an **opt-in** accessibility aid, constrained by two principles: presentation-neutral content (§1.6) and "nothing leaves the device" (§1.2).

- **Web Speech API only** (`speechSynthesis` + `SpeechSynthesisUtterance`) — the engine already in the browser/OS. No audio assets, no TTS service, no network call.
- **On-device voices only.** The footer Voice picker lists only `localService` voices, so Explanation text is synthesized locally and never sent anywhere. A cloud voice would silently break the privacy model.
- **Opt-in, off by default.** Nothing renders until the reader picks a **Voice** in the footer; the choice persists per browser under `quizbun.voice-uri`. No surprise audio, no read-aloud chrome for readers who never opt in.
- **Speaks flattened text**, not rendered markup — the utterance reads the two-tier pipeline's text output (Markdown/HTML stripped, entities decoded), so the engine never voices `**`, code fences, or link URLs.
- **English-only in v1**, filtered to the BCP-47 primary subtag `en`, matching the English-only site chrome.
- **Cross-island shared store.** The footer (Voice picker) and the player (read-aloud control) are independent Astro islands with no common React tree, so the selection lives in an external store (`src/shared/lib/speech`) consumed via `useSyncExternalStore`, synchronized through three browser-global channels: `localStorage` (persisted truth), the `voiceschanged` event (the async-loading voice list), and a custom `quizbun:voice-preference` window event (same-tab updates, which the native `storage` event deliberately does not deliver to the originating tab).

Cloud TTS and autoplay/always-on reading were both rejected — the first leaks text off-device, the second is a hostile accessibility anti-pattern. The feature is small, additive, and reversible: remove the footer picker and the control disappears, with no schema or storage migration.

### Decision: popup stacking model (Base UI portals)

Base UI renders every popup (Dialog, Popover, Select, Combobox) into a portal appended to `document.body` with no inline `position` or `z-index` — the portal div is `position: static`. Layering all portals above page content, and above each other in a stable order, rests on two mechanisms:

- **One shared rule**, `[data-base-ui-portal] { z-index: 2 }` in `global.css`. This only takes effect because `layout.astro`'s root `<body>` is `display: grid` — grid (and flex) items honor `z-index` even while `position: static`; a plain block-level sibling would not. **If any layout stops using a grid body, this rule silently becomes a no-op** and portals fall back under page content.
- **DOM append order as the cross-portal tiebreak.** Portals mount only while open, so whichever popup opened most recently is the last portal child of `body` and paints last (highest) among elements that all share `z-index: 2`. This is what stacks a Combobox's overflow popover above its own dropdown, and what would layer a Select opened from inside a Dialog above the Dialog.

**`z-index` on the popup elements themselves cannot do this job.** Every anchored popup (Popover, Select, Combobox) is positioned by a floating-ui positioner that receives an inline `transform: translate(...)`; Dialog centers itself the same way (`position: fixed; transform: translate(-50%, -50%)`). A `transform` on a positioned, z-indexed element always creates its own stacking context, which seals any `z-index` declared on the popup or its descendants away from comparison with a *different* portal's popup. Two dead declarations were found and removed for exactly this reason: `.popup` in `combobox.module.css` (commit `10d0aeae`) and `.overflow-popover` in the same file (commit `bc96a55a`), which also replaced the z-index-equality assertion in the combobox layering test with `expect.poll` + `document.elementFromPoint` hit-testing. **Assert visible paint order in browser tests, never a computed `z-index` value.**

`dialog`, `popover`, and `select` still set `isolation: isolate` / `z-index` next to their own popup markup (e.g. Dialog's backdrop below its popup, Select's highlighted-item overlay). These remain legitimate *only* for ordering elements within one portal's own stacking context — they are not evidence that cross-portal ordering happens there.

Page content must keep any unscoped `z-index` below `2`. `import-quiz-form.module.css` needs its own `z-index: 2` internally (a sticky actions toolbar) and traps it with `isolation: isolate` on `.root` so it can never be compared against a portal's `z-index: 2`.

**Proposal, not adopted:** Base UI's own docs recommend an `isolation: isolate` wrapper placed *inside* `body`, with portals rendered as siblings of that wrapper, which makes every page `z-index` irrelevant to popups by construction. Quizbun instead sets `isolation: isolate` on `body` itself, which does not give that guarantee — portals and page content still share `body`'s own stacking context. Adopting the wrapper would remove the grid-body dependency above, but is a real markup/CSS change (an added wrapper element and portal boundary); evaluate it as its own change rather than folding it into an unrelated one.

## 5. Surfaces, routes, and behavior

Private routes use **static shells plus query params** — no hash router, no `404.html` fallback. GitHub Pages serves only build-time files; a query param is the honest way to address content that exists only on the visitor's device.

| Route                                                             | Kind             | Surface                                               |
| ----------------------------------------------------------------- | ---------------- | ----------------------------------------------------- |
| `/`                                                               | static + island  | Home: marketing + recently added + continue block     |
| `/quizzes/`, `/quizzes/page/{n}/`                                 | static + island  | Catalog list (Tag filter, `?tags=` in URL), paginated |
| `/quizzes/{id}/`                                                  | static, per quiz | Public quiz detail + player island                    |
| `/library/`                                                       | static shell     | Library list (client-rendered from IndexedDB)         |
| `/library/quiz/?id={id}`                                          | static shell     | Private quiz detail + player                          |
| `/import/`                                                        | static + island  | Import page                                           |
| `/docs/` (+ `standard/`, `prompt/`, `examples/`, `contributing/`) | static           | Docs surface                                          |
| `/docs/examples/{file}.json`                                      | static endpoint  | Raw canonical example JSON (download)                 |
| `/schema/quiz.v1.json`                                            | static asset     | Published JSON Schema                                 |

### Home and the Catalog (the split)

Home (`/`) is a **marketing page**, not the Catalog. It carries: what Quizbun is and why (copy derived from the product vision), the **10 most recently added** Catalog quizzes, and — when the browser has any — up to **5 unfinished Runs** to continue. The Catalog list itself lives at `/quizzes/` with the Tag filter and `?tags=` in the URL; the header's "Quizzes" item points there, and the logo links Home.

- **Recency** is derived at build time from git: the first-commit date of each file in `content/quizzes/` (`addedAt` on the summary projection, ISO 8601). The Standard is untouched — no date field. A file with no git record (uncommitted in dev, or a shallow checkout) falls back to "added now"; the deploy therefore checks out with full history (`fetch-depth: 0`), and the build warns if more than half the quizzes hit the fallback. Ties (a whole seed cluster committed together) break by filename for deterministic builds.
- **The continue block** is a client-only island (IndexedDB has no build-time existence): it lists unfinished Runs across both namespaces, newest activity first via `updatedAt`, takes the first five *resolvable* Runs, and renders nothing at all when none resolve. Catalog Runs resolve against build-time summaries serialized into the island; Library Runs resolve against the IndexedDB quizzes store; a Run whose quiz is gone is skipped, never shown broken.
- **`?tags=`** is a single comma-separated param (`?tags=css,web`), synced with `history.replaceState` (no history-stack spam). Tags are constrained to the kebab charset, so commas are safe. Unknown tags in the URL are silently ignored — they may belong to a removed quiz.

### The player

The player is the heart of the loop, hosted identically on the public `/quizzes/{id}/` pages and the private `/library/quiz/` shell; the quiz object is injected (from build-time props or from IndexedDB) rather than assumed to come from one source.

- **Pagination by Page size** (`1 | 3 | 5 | 10`, default 5, global localStorage), adjustable mid-Run. Re-chunking loses nothing because answers key by Question `id`, never by page.
- **Per-Question lifecycle:** submit → correctness + Explanation, then optional References, shown in place → the Question locks for the Run (controls disabled, answer visible). Auto-save fires on every submit, so a reload never loses a submitted answer.
- **Free navigation:** pages in any order. A Question is either submitted (locked) or unsubmitted (answerable whenever) — there is no "skipped" state.
- **Resume:** reopening an in-progress Run lands on the first page containing an unsubmitted Question.
- **Finish → Summary:** Finish appears only when every Question is submitted; the Summary shows "X of Y correct," a per-Question correct/incorrect list linking back to each Explanation, plus **Retake** and **Back**. Retake replaces the Run (no archive).
- **Question components:** `single-choice` (radio group), `multiple-choice` (checkbox group, all-or-nothing), `input` (text/numeric via the shared `check-answer` module). Options render in original JSON order in v1; Option text renders through the inline tier, prompt/Explanation/References through the full tier.
- **Read aloud** (§4) attaches a per-Explanation speak control once a Voice is chosen.

### Import, Library, and quiz detail

- **Import (`/import/`):** one `<textarea>` is the single input surface — file upload and drag-and-drop just fill it. Parse JSON → validate against the Zod schema → on failure render the formatter output verbatim (path-precise, ordered, copyable as one block; JSON syntax errors get the same human-readable treatment) with the textarea still editable → on success preview (title, question count, tags) and save. An `id` collision triggers the explicit replace-or-cancel dialog, stating what will happen to Progress; replace runs the reconciliation above.
- **Library (`/library/`):** a static shell whose island lists quizzes from IndexedDB — list, import, open, **Export** (the stored Quiz JSON exactly as imported; Progress is never exported), delete (confirms; cascades the Run), and the Tag filter (the same data-source-injected component the Catalog uses — it never knows about IndexedDB).
- **Quiz detail** (shared by public and private surfaces) hosts title, description, tags, author, question count, Export, Reset progress, and a **state-aware primary action** driven by Run status: **Start** (no Run) / **Continue — X of Y answered** (in progress) / **See summary** + **Retake** (finished). Activating it swaps the player in on the same route — no navigation. Reset progress and Retake are the same underlying operation (delete the saved Run).

## 6. Public catalog and contribution pipeline

- **Catalog content** lives as one JSON file per quiz in `content/quizzes/*.json` — outside `src/` so contributors never touch app code, outside `public/` so files are not shipped twice. Filename must equal the quiz `id`. The build-time loader (`src/shared/lib/content/`) validates every file through the Zod schema (a bad file fails the build through the formatter), and treats duplicate ids or filename ≠ id as **build errors**, not merely CI errors.
- **The Catalog currently ships more than 40 public quizzes** across many topic clusters (JavaScript/TypeScript, CSS, Node.js internals, systems/networking, economics, and more). All three Question types and both `input` modes are represented, and tags overlap across clusters so the filter is genuinely useful. (This is well beyond the original ~8-quiz seed target — the seed pipeline succeeded and the Catalog kept growing through the normal review flow.)
- **CI profile validator** (`scripts/validate-public-quizzes.ts`) imports the loader (base Standard validation comes free), then applies the Public catalog profile and Markdown/sanitization checks. Failures print through the same formatter as the import page and end with a link to the contributor guide's error-round-trip section, so a rejected contributor lands on the page that explains what to do.
- **Contribution surface:** the docs section publishes the standard document, a copyable prompt page, the canonical examples (with downloads), and a contributor guide (`docs/contributing/`) covering generate → validate on `/import/` → meet the profile → open a PR → read CI feedback → revise. A single PR template mirrors the profile and review focus; a root `CONTRIBUTING.md` is a pointer, not a fork. The MIT `LICENSE` at the repo root backs the license-by-contribution statement (submitting a public quiz licenses it under the repo license; there is no per-quiz license field).
- **Human review** — beyond CI — evaluates topic quality, clarity, and Explanation value. An Explanation that merely restates the correct answer fails review; explanation-first is the product.
- **The error-message round-trip is documented as a feature:** the contributor guide shows a real broken quiz, the genuine formatter report it produces, and that report pasted back into an AI chat to get a corrected file — the same report shape on both the import page (creator) and the CI check (contributor).

## 7. Testing and CI

Tests run on **Vitest**, in two in-repo lanes plus a separate end-to-end lane:

- **Unit** (`.spec.ts`, Node environment) and **component** (`.test.tsx`, real Chromium via `vitest-browser-react`) projects under one `vitest.config.ts`. The Standard's fixture-based `valid/`/`invalid/` set is the regression net; the error formatter is covered by snapshot tests (its output is an API, so changes show up in diffs).
- **End-to-end** (`e2e/`, `.e2e.ts` suffix) on **Playwright**, driven against the static `astro preview` build with **relative** navigation. Library journeys seed IndexedDB directly via `e2e/fixtures/seed.ts`; the import journey is the one that exercises the real UI; Catalog journeys need no seeding because public quizzes ship in `dist/`. Each test gets a fresh browser context (empty IndexedDB + localStorage). The core journeys are covered — import→run→finish, Catalog browse/filter, all Question types, resume, Library management, Page-size re-chunking, Content-hash invalidation, and keyboard/phone/theme passes. One optional deploy-fidelity spec (base-path deep links under `GITHUB_PAGES=true`) remains open.

**CI** is a single `.github/workflows/ci.yml`. Its check set: `css:dts` (regenerate CSS Module typings so the typecheck sees fresh class names), `typecheck`, `check:astro` (`astro check`, covering `.astro` template diagnostics `tsc` doesn't see), the unit and browser-component Vitest suites (`bun run test`, with Chromium installed in CI), `check` (oxlint + oxfmt with import sorting), `stylelint`, `validate:docs-examples`, `validate:public-quizzes`, the JSON Schema drift check, `astro build`, and a `dist/` base-path guard (`scripts/check-dist-base-paths.sh`). A `deploy` job (`GITHUB_PAGES=true bun run build` → upload → `actions/deploy-pages`) gates on the checks passing.

> **Current CI trigger state:** while the repository is private, the automatic `push`/`pull_request` triggers are commented out; CI and deploy run via `workflow_dispatch` only (with a `deploy` boolean input). The non-blocking `e2e` job is dispatch-driven too. Re-enable the automatic triggers when the repo goes public — that also unblocks verifying the profile check on PRs from forks.

## 8. Non-functional requirements

- **Browsers:** evergreen only (current Chrome, Firefox, Safari, Edge). IndexedDB and modern CSS assumed.
- **Mobile:** fully responsive; the player is designed phone-first.
- **Accessibility:** semantic HTML, full keyboard operability (the player is completable without a mouse), visible focus states, labeled form controls, `fieldset`/`legend` per Question, live-region announcement of correctness on submit, focus trapped and restored in dialogs. WCAG AA is the target, not a certified audit gate. Read aloud is an additive aid on top of this baseline, never a substitute for it.
- **Performance:** the architecture *is* the strategy — static pages, islands only where interactivity exists, and no client-side rendering of build-time-known content. No numeric budget is chased in v1.
- **Theming:** `light-dark()` with `<html data-theme>` is the only theming mechanism; components consume semantic tokens only (never raw palette values). CSS Modules are co-located; conventions are documented and enforced by the `css-modules-quizbun` skill (`.claude/skills/css-modules-quizbun/SKILL.md`).

## 9. Release status and what remains open

- **Release model:** continuous deployment is the intended model — every merge to `main` deploys to GitHub Pages, no release ceremony. v1.0 = the creator/contributor surface complete **and** the Standard frozen, in one tag.
- **Frozen:** the `v1.0.0` tag exists on the release commit; `package.json` is at `1.0.0`; the draft banner is gone from `docs/standard.md` and replaced with the frozen/versioning statement; the pre-freeze sweep found no schema or error-message change warranted (all public quizzes validate at zero warnings, `schema:check` reports no drift). From here, breaking changes require `schemaVersion: 2`; adding optional fields is the only in-version evolution.
- **The one open verification:** everything above has been exercised on local production builds (`bun run build` + `astro preview`, including a `GITHUB_PAGES=true` build served under `/quizbun/`). What has **not** happened yet is a real GitHub Pages deployment and the acceptance run against the *live* site — the deploy is deliberately deferred while the repo is private. When the repo goes public: push `main` and the tag, publish the GitHub Release from the drafted notes, re-enable automatic CI triggers, run the end-to-end acceptance flow on the live URL (browse → filter → complete a Run → export → re-import as a private copy → confirm both namespaced Runs coexist), and verify the CI profile check fires on a fork PR.

## 10. Deferred to version 2 (out of scope for v1)

Each cut is intentional, with its reason — recorded, not dropped, so a future planning round stays honest. Adding these later is additive; none was blocked by a v1 mistake.

- **Progress export/import across devices** — needs a second versioned standard plus merge semantics; the quizzes themselves are already portable.
- **Attempt history / statistics** — a new data model and stats UI; pairs naturally with progress export.
- **Partial-credit / scoring** — exam machinery that contradicts learning-first; an optional `scoring` field could be added later.
- **Raw HTML in content** — every real case is covered by Markdown; allowing it now and restricting later would be a breaking change.
- **Per-quiz license / structured author metadata** — the repo license plus an optional `author` string is enough.
- **PWA / offline support** — its own project; "no servers" holds regardless.
- **UI internationalization** — site chrome ships in English; quiz *content* can be any language via `language`.
- **Option-level ids, presentation hints, richer taxonomy** — all additive if ever genuinely needed. (A verbose taxonomy was the main mistake of the first attempt; tags-only is a deliberate hard constraint.)
- **Length caps** on Standard fields — deliberately omitted (§3 micro-decision 3), because tightening later is breaking.
- **Example "try it" links** into the player from the docs — would require loading a known quiz into the private player shell (a new feature); the Catalog already demonstrates the player.
- **Reading Explanations in non-English voices** — waits until the Renderer can match a Voice to the quiz `language`.

## 11. Risks

- **Schema mistakes discovered late** — mitigated by having exercised the Standard hard through the local loop, the Catalog build, and 44 quizzes' worth of generation before the freeze; the pre-freeze sweep found none warranting a change.
- **Live-deploy surprises** — the only unexercised path is the live Pages deploy (§9). Base-path handling is guarded at build time (`check-dist-base-paths.sh`) and verified on a base-mounted preview, which reduces but does not eliminate the risk; the live acceptance run closes it.
- **Contribution friction from strictness** — countered by treating error-message quality as a first-class feature (the paste-back round-trip) rather than a polish task.
